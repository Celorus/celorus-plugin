# Plugin & Embedded Surfaces

Constrained design guidance for Celorus UI that renders inside someone else's chrome: Claude/Cowork plugin views, embedded dashboards, iframes, chat-rendered widgets. On these surfaces the host owns the window, the fonts may not load, and the viewport may be 360px wide. The brand must survive anyway.

## Two tiers — decide FIRST which one you are on

Not every hosted surface is "embedded":

- **Tier 1 — full-page hosted artifact.** The page owns the whole browser tab even though a
  host serves it (the claude.ai artifact viewer, any hosted single-file HTML report or
  dashboard). Treat it as a **standalone page**: charcoal topbar, raised sheet, full section
  rhythm, editorial type scale — the standalone skeleton applies unchanged. Fonts: these
  hosts' CSP blocks font CDNs, so inline `tokens/dist/embedded-fonts.css` (data-URI subsets,
  latin + ₹) instead of relying on `@import`. Do NOT apply the compact contract below.
- **Tier 2 — truly embedded.** The surface shares the window with host chrome: plugin panels,
  iframes, cards in a feed, chat-rendered widgets, anything that can be ≤480px wide.
  Everything below this line is the Tier-2 contract.

## The contract (Tier 2)

1. **Consume semantic tokens only.** Embedded views import `tokens.css` (or inline the foundation block) and bind exclusively to `--semantic-*` / `--component-*` vars. Never restyle per-host.
2. **Theme follows the host.** Detect with `prefers-color-scheme` and honor an explicit `data-theme` if the host sets one. Both themes ship in `tokens.css`; an embedded view must look correct in each without per-view code. Rationale: a plugin that forces bone-light inside a dark host reads as broken, not branded.
3. **Fonts degrade gracefully.** Web-font loading inside plugin sandboxes is not guaranteed. The token font stacks already carry fallbacks (Georgia / system-ui / ui-monospace). Do not fight this — the palette, spacing, and eyebrow pattern carry the brand when the serif doesn't load.
4. **The host owns the chrome (Tier 2 only).** No charcoal topbar, no sticky nav, no progress bar in truly embedded views. Identity comes from one wordmark or hexagon mark placed once (footer or header corner), never repeated. (Tier-1 full-page artifacts keep their topbar — see above.)
5. **Provenance still ships.** Any emitted artifact keeps its provenance line (`Generated by Celorus · date · sources`). This rule has no exceptions anywhere in the system.

## Small viewports (≤ 480px) and dense hosts

- Single column, always. Grids (`.c-form-grid`, stat tiles, signal grids) collapse to `1fr`.
- Compact spacing: step paddings down one level (`--semantic-space-*`: use 8/12 where full surfaces use 16/24; section padding 24px, not 60px).
- Tables become key–value stacks below 480px; keep mono for values, right-align nothing.
- Touch targets ≥ 44×44px for any tap-able control regardless of visual size.
- Type floor: body never below 13px; labels never below 10.5px; hero/serif display sizes are not used in embedded views (h2 max 22px).
- Wordmark below its 90px minimum → use the favicon mark alone (see `brand/logos/ICON_SIZES.md`).

## Do / Don't

| Do | Don't |
|---|---|
| Inline the foundation block for sandboxed iframes | Link cross-origin CSS you don't control |
| One accent: petrol (light) / `#3FA6BB` step (dark) via tokens | Adopt the host's accent color "to blend in" |
| Use `.c-` components at `sm` sizes | Ship bespoke one-off controls per surface |
| Keep focus rings — hosts don't provide them for you | Remove outlines because "it's just a widget" |
| Empty/loading/error states from the component library | Blank panels while data loads |

## Minimum viable brand (when everything is constrained)

If a surface allows almost nothing (e.g., markdown-only chat output), the brand is carried by: mono-formatted data values, the eyebrow-style uppercase label convention, provenance line, and operator voice (`brand/VOICE.md`). No color is better than wrong color.
